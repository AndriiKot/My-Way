Правила соглашения о коммитах (Conventional Commits)
Структура сообщения коммита:

Формат: <type>[optional scope]: <description>
Пример: feat(core): add new feature
Типы коммитов:

feat: новая функция (feature)

fix: исправление ошибки

docs: изменения в документации

style: изменения, не влияющие на логику (пробелы, форматирование и т.д.)

refactor: изменения кода, не исправляющие ошибок и не добавляющие функций

perf: улучшение производительности

test: добавление тестов или исправление существующих

chore: изменения, не относящиеся к приложению (например, обновление зависимостей)

build: изменения, касающиеся сборки или внешних зависимостей

Обязательная часть:

description — краткое, но информативное описание изменений.
Breaking Changes:

Если ваш коммит является нарушением обратной совместимости, вы можете указать это в сообщении,
добавив BREAKING CHANGE: в конце сообщение или используя ! перед двоеточием после типа и области.
Примеры:

feat: add a new authentication method
fix: resolve issue with user login
docs: update README with new installation instructions
chore!: drop support for Node 6

Последовательность и консистентность:

Соблюдайте единообразие в формате 
ваших сообщений коммитов для улучшения читаемости.
